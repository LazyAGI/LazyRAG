需求清单:

1. 基于真实的场景试用我们的能力，完善既有功能和使用体验：
  - chat界面新增mermaid的输出和展示，渲染失败时展示源码
  - 知识库标签需要限制长度。
  - 修复文档 block 切片断句问题。
  - 知识库上传文档解析并导入时，需要提示用户不要关闭浏览器（也可能是标签页），并在上传完成立即退出上传界面，无需等word2pdf。
  - 问答场景下，知识库选择 UI 需要优化，支持最近使用和收藏，支持筛选，支持设置默认偏好，支持"全部"。
  - 缩短知识库页面刷新时间。
  - 删除知识库时增加确认信息，并检测知识库中是否存在内容或关联数据。
  - 知识库中的文档支持修改切片名称，包括 line 和 block。
  - 知识库的文件支持分类整理，知识库文档支持移动到其他文件夹或知识库，不支持复制。
  - 邮箱长度限制需要缩小。
  - 用户角色变更时，需要限制 system-admin 的角色变更规则。
  - system-admin 不能被禁用
  - 知识源添加的时候，后端可以扫本地盘 / 飞书的目录结构，供用户在前端展开和选择
  - P1：Diff 差异对比 UI 升级，用户能够清楚看到行内具体修改了什么。支持 AI 总结变更点。
  - P1：记忆模块版本管理，支持回退；支持回收站。
2. 新增 Personal Mode：
  1. 仍然有用户的概念，只有一个用户组（默认直接创建，并让admin加进去），免登录，进去就是admin，可以新建用户
  2. Personal Mode去掉RBAC和kong gateway，postgres和opensearch换sqlite，milvus换milvus-lite
  3. 用Electron包一下，把docker去掉，后端（go和python）预编译成二进制打包，不带开发环境；然后直接在Electron中启动前后端
  4. Personal模型下，"新建用户"显示为"新建AI助手"，实际上是后台就是用户。新建的用户加入到默认的用户组中，权限为"写"。技能页面可以直接切换用户，看该用户的技能；问答页面也可以直接切换用户，以该用户的身份进行问答。一些前端程序，也要对应做修改。
  5. （P1）给每个用户加一个事件循环，用于监听任务；接入微信 / 飞书
3. 算法自进化在至少一个数据集上取得效果，给出实验报告，包括基础算法，友商方案和自进化后的算法效果对比
4. Key管理与模型配置：
  1. API Key加密存储，前端永不回显完整 key
  2. 增加embedding和MinerU / PaddleOCR API Key的配置策略（涉及离线解析，是否用户各陪各的待定）。
  3. 增加部分核心工具的key配置策略，如搜索引擎api key等
  4. 用户组管理员可以选择把配置共享给成员（模型这边，不是共享的供应商管理，而是共享的模型配置）。此时优先使用成员自己配置的，如果没有，再选用管理员配置的。在Personal模式下，默认开启共享配置
5. 支持部分功能配置为可选
  1. embedding可选，具体的：
    - 如果需要向量，则初始化向量库，稠密向量，稀疏向量至少选一个
    - 如果不需要向量，那么会使用bm25进行召回，此时向量库将不再初始化
  2. rerank可选，如果不选，则不使用rerank模型
  3. paddleocr和mineru可选，如果不选，则word2pdf也不做
  4. 多模态模块可选（vlm，sd，stt，tts，...），如果不选，后端直接限制相关功能
6. 问答算法功能增强和代码简化：
  1. 优化chat和agentic-chat的代码逻辑，目标是代码量减少70%，逻辑清楚，算法和工程分离，可复用能力沉淀到lazyllm中
  2. 都变成从"包"的相对引用，而不是绝对路径引用。此外相对路径的顶层（chat，parsing）问题比较大，应变成lazyrag，即from lazyrag.algorithm.chat import xxx
  3. 删掉非流式输出的相关内容，由后端将流转成非流（此举也可以避免非流模式下后端-算法超时）
  4. 优化agentic工具的注册和选取逻辑，支持运行时传key
  5. 增加多用户下的key配置逻辑，增加部分功能在无key时的跳过逻辑
  6. 支持动态搜索飞书知识，在此场景下，配置好飞书知识源之后，不入库，只使用key做动态查找和修改
7. 数据集管理：
  - 新增数据集管理模块（考虑要哪些字段，比如query，React-chain，reference，answer？），支持用户上传，和从选定的知识库的问答记录中回流
  - 支持对数据进行新增和在线修改，每个数据都会标记出处（用户上传，回流，手动）
8. 算法自进化：
  1. 优化控制体验，前端呈现和checkpoint机制完善
  2. 开发"新算法的上线和切流"的机制。具体的：
    - 一个知识库可以配置多个算法，这些算法都会被离线解析所用到。普通用户对算法不可见，管理员在开发者模式下，可以看到算法。
    - 在chat的时候，有一个"router模块"用于调控chat究竟选择哪个算法，或者是双回复。默认情况下，会选择主算法。
    - 自进化模块修改完代码后，需要再沙箱里面跑通单元测试（不污染主环境），然后进行冲突检查，如果没有冲突，则将算法注册到主服务中，并且向基础知识库+被测知识库加入该算法，并发起部分重解析（增量解析多的ng）。评测后，如果算法效果有提升，则将所有的知识库都新增该算法，然后进入灰度测试阶段。
    - 灰度测试即配置router，对指定的知识库进行切流；灰度通过后，将新算法设置为主算法。30天内支持回滚，过期删除旧的算法。
  3. 新增数据集管理使得自进化的时候能够选择既有的知识库和测试集
9. 智积阅累：
  1. 提供专门的入口，允许配置AI的角色，回复风格和对你的称谓（会直接存放到记忆中）
  2. 技能/记忆/习惯/偏好自动更新配置。具体是，在技能、记忆、习惯、偏好设置中增加「是否自动更新」配置，原「保护态」相关文案全部调整为「是否自动更新」，开启自动更新后，系统可以自动修改对应内容，用户无感知更新。关闭自动更新后，系统只能提出修改建议，需要用户人工审批后才可生效。词表无法配置，默认自动更新，有冲突时提醒用户处理；技能默认不自动更新；记忆、习惯、偏好默认自动更新。
  3. 接受建议后立即跳到下一个界面，并提示用户等待计算；每个文档同时只有一个task，重进文档的建议页面恢复 task。具体的，用户点击接受建议后，需要跳转到现有的生成草稿界面。页面刷新后，需要恢复当前待处理 task。task 状态需要能够从后端或持久化状态中恢复。当task被明确应用或放弃后，才能创建新的 task，如果task执行失败，后端会帮助用户重试 。过往的处理记录可以在日志中看到。
  4. 优化记忆的更新策略和触发条件，增加"跨文件整理"功能。
  5. 增加时间权重，当用户偏好发生变化时导致记忆冲突时，根据时间权重重新整理记忆。
  6. 确保总结的内容的通用性和有效性
  7. 避免重复总结，一个key-point只能出现在技能，词表，记忆之一
  8. 支持直接从数据集中学习技能，词表和记忆
  9. P2：React + Skill 引入确定性机制和人工介入节点
10. 权限：参考 LazyRAG设计的 3.5 权限模型，使用和授权方式如github。Personal模式下，机器人用户默认是write权限
11. 支持使用github，google，飞书等账号登录。一个账号可以关联手机号，邮箱，github，google，飞书等多个账号，如果同时绑定多个，任一登录方式均可。
12. P1：新增工具管理：
  1. 支持用户配置mcp工具和restful-api工具，暂不支持code工具和cli工具
  2. 支持在智积阅累的工具模块配置工具的开启和关闭，包括a中新增的工具
  3. 支持在skill中引用工具。如果发现引用工具被删除，则应忽略，并提醒用户删除引用。

16. P2：浏览器插件：写个浏览器插件，在PersonalMode下，用户安装了咱们的项目之后，就可以通过这个插件跟我们的服务进行chat，会把当前的页面的关键信息带进去chat。对于我们支持的能力，比如飞书等，可以直接发出阅读和修改的指令；特别的，如果用户没有配置oauth认证，告诉他要先配置认证。对于我们不支持的能力，只读，如果用户想修改，告诉他不支持。下面是一些例子：
  - 用户可以打开一个飞书页面，召唤出我们的对话框，说"帮我填一下周报"：
    - 如果用户配置过oauth，我们的agent就会自动拿到这个页面的url或者id，进一步的，按照用户给定的"周报skill"，分析知识库和本地文件，看看用户这周都干了什么，然后帮用户把信息填上。
    - 如果用户没配置，我们调用工具失败，会告诉他如何配置
  - 用户打开了淘宝页面，我们不支持修改，只读。
    - 如果用户问：这个商品评价怎么样，我们会把页面内容抓到，却分析什么商品，联网搜索评价。
    - 如果用户问：帮我下单，我们会告诉他，我们做不到，只能读，不能操作。
